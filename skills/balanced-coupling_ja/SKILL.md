# Balanced Coupling（均衡結合）モデル

Vlad Khononov による Balanced Coupling モデルの包括的なリファレンス。このドキュメントは [coupling.dev](https://coupling.dev) のブログから全モデルを統合したもので、モデルの基礎、各次元、バランスのメカニクス、従来の結合モデルとの関係、そしてドメイン駆動設計との接続を網羅する。

---

## 1. 基礎: なぜ結合（Coupling）が重要か

### 1.1 結合は敵ではない

「結合（Coupling）」という言葉は長らく悪い設計の代名詞とされてきた。しかし Balanced Coupling モデルはこの前提に異議を唱える。辞書的には、結合とは単に部品をつなぐ仕組みのことだ。システムが目標達成のために相互作用するコンポーネントの集合であるなら、結合はそれらのコンポーネントをつなぎ、目標達成を可能にするものだ。**結合こそが、システムの価値を部品の総和以上のものにする。**

完全に独立したコンポーネントでシステムを構築することはできない――それは「システム」の定義に反する。問題は結合するかどうかではなく、*どのように* 結合するかだ。ある種の結合はモジュラリティをもたらし、別の種は複雑性を生む。Balanced Coupling モデルは、結合を排除すべき厄介者ではなく、意図的に使いこなすべき**設計ツール**として扱う。

### 1.2 複雑性（Cynefin フレームワークによる定義）

このモデルは Cynefin フレームワークの複雑性の定義を採用する:

- **明確（Clear / Simple）**: 変更を加えたとき、結果が正確にわかる。
- **煩雑（Complicated）**: 結果はわからないが、専門家はわかり、相談できる。
- **複雑（Complex）**: 結果を知る唯一の方法は変更を加えて観察すること。原因と結果は事後的にしか特定できない。
- **混沌（Chaotic）**: 行動と結果の間に、事後的にも特定可能な関係がない。

ソフトウェア設計において、**複雑性**とは変更の結果が予測不可能なシステムを意味する。モジュラ設計の目標は、変更の影響が事後的にしか把握できない状況を避けることだ。

重要なのは、**複雑性は主観的**であるという点――それはシステム設計と私たちの認知的限界の両方に依存する:

```
complexity = f(system design, cognitive limits)
```

研究によれば、人間が作業記憶に同時保持できる情報単位は 4 ±1 個とされている。変更に必要な情報がこの認知容量を超えると、設計は複雑になる。

### 1.3 モジュラリティ（Modularity）: 複雑性の対極

複雑性が変更の結果を予測不可能にするなら、モジュラリティはその対極だ。モジュラ設計は2つの基準を満たす:

1. システムのどの部分を変更すべきかが**明確**である。
2. 変更の**結果が明確で予測可能**である。

モジュラリティはシステムの目標を未来へと延伸する。これは将来の要件を先取りして実装することを意味しない――それは不可能だ。そうではなく、*妥当な* 将来の変更実装を、必要な認知的負荷を最小化することで容易にするということだ。

システムはコンポーネントが目標達成のために相互作用するものだ。「大きな泥の玉（Big Ball of Mud）」でさえ現在の目標を達成できる。モジュラ設計はシステムの目標を、目標が進化しても*継続して*達成できるよう拡張する。

### 1.4 共通の根源

複雑性とモジュラリティは対照的な概念だが、同じ根本的な原因に動かされている: **結合**だ。コンポーネントの接続方法が、システムがモジュラになるか複雑になるかを決定する。

## 2. 結合の3つの次元

2つのコンポーネントが結合しているとき、モジュラリティか複雑性かを決定するのは、コンポーネント自体ではなく、それらの間の**接続**だ。Balanced Coupling モデルはこの接続を **3つの次元** で評価する: 統合強度（Integration Strength）、距離（Distance）、変動性（Volatility）。

### 2.1 統合強度（Integration Strength）: 共有される知識

統合にはコンポーネントが互いに**知識**を交換する必要がある。共有される知識が多いほど、一方のコンポーネントの変更が他方への連鎖的変更を引き起こす確率が高くなる。

知識を定量的に「測定」しようとするのではなく、このモデルは知識の種類を**分類**する。各種類には大きく異なる量の共有知識が伴う。これが **Integration Strength（統合強度）**モデルであり、4つのレベルを特定する（強い/多知識から弱い/少知識へ）:

#### Intrusive Coupling（侵入的結合）— 最強
**プライベートなインターフェース**や実装の詳細を統合に使う場合に発生する――内部データベース、プライベートオブジェクト、非公式 API など。コンポーネントの実装に関する**すべての知識**が共有されていると見なす必要がある。侵入的結合は**脆く**、かつ**暗黙的**だ――「侵入された」コンポーネントの作者は統合の存在すら知らない場合がある。

古典的なモジュール結合モデルの **content coupling（内容結合、病的結合）** に相当する。

#### Functional Coupling（機能的結合）
実装の*方法*から、実装している*内容*――つまり**機能要件**へと焦点を移す。複数のコンポーネントがビジネス要件の知識を共有し、それらの要件が進化するときに一緒に変更しなければならない場合に発生する。

極端なケースは**重複した知識**だ――同じビジネスルールがフロントエンドとバックエンドの両方に実装されている状況。仕様変更が両方に同時に反映されなければ、システムは不整合な状態になる。

侵入的結合と同様に、機能的結合は**暗黙的**になりえる: 明示的に結合されていないコンポーネントに重複したビジネスロジックが存在し、それでも一緒に変更する必要がある。

古典的なモジュール結合モデルの **common coupling（共通結合）、external coupling（外部結合）、control coupling（制御結合）** に相当する。

#### Model Coupling（モデル結合）
コンポーネントが**ビジネスドメインモデル**の知識を共有する場合に発生する。モデルが変わると――新しいドメインの洞察により――結合したすべてのコンポーネントが変更しなければならない。

ドメイン駆動設計は、一つの巨大な包括モデルではなく、複数の専門化されたモデルを扱うことを強調する。モデル結合は、コンポーネントがこれらのモデルの一つを共有するときに生じる。

古典的なモジュール結合モデルの **stamp coupling（スタンプ結合）** に相当する。

#### Contract Coupling（コントラクト結合）— 最弱
共有される知識の最低レベル。統合コントラクト（Integration Contract）は実装の詳細、機能要件、ビジネスモデルを**カプセル化**し、統合を明示的かつ安定したものにする。

コントラクトを導入する設計パターン: Facade（ファサード）、Open-host service / Published language（DDD）、Anti-corruption layer（DDD）、Data transfer objects（DTO）。

古典的なモジュール結合モデルの **data coupling（データ結合）** に相当し、内部モデルと明示的な統合コントラクトを区別することで精緻化されている。

#### 暗黙-明示スペクトラム

4つの統合強度レベルは、共有知識がどれほど**暗黙的または明示的**かも反映している:

- **Contract Coupling** = 最も明示的
- **Model Coupling** = 明示的だが範囲が広い
- **Functional Coupling** = 暗黙的になりえる
- **Intrusive Coupling** = 最も暗黙的で脆い

#### 統合強度の度合い

各レベル内では、**connascence（コナセンス）** のレベルを使ってより細かい比較が可能だ:
- **静的コナセンス** のレベル（名前、型、意味、アルゴリズム、位置）は Contract および Model Coupling の度合いを比較できる。
- **動的コナセンス** のレベル（実行、タイミング、値、同一性）は Functional Coupling の度合いを比較できる。

### 2.2 距離（Distance）

結合したコンポーネント間の物理的・論理的距離は、**変更のコスト**に影響する。同じファイル内の2つのメソッドを共に進化させることは、異なるマイクロサービスや別システムの2つのオブジェクトを共に進化させることよりも大幅に容易だ。

距離は抽象化のレベルに応じて増大する: メソッド → オブジェクト → 名前空間/パッケージ → （マイクロ）サービス → システム。各レベルで、同時変更の認知的努力と調整コストが増大する。

#### 距離は観察する抽象化レベルに相対的

距離は絶対的な尺度ではなく、**分析する抽象化レベルに相対的**だ。このモデルはフラクタルだ: 同じバランスルールが、クラス内のメソッド間でも、モジュール内のクラス間でも、サービス内のモジュール間でも、システム間でも適用される。

任意の抽象化レベルにおいて、**最大距離はそのレベルの境界**だ。プロジェクトが複数モジュールを持つ単一サービスで構成されている場合、それらのモジュール間の距離がそのコンテキストで最大距離になる。同じ原則が適用される: 2つのモジュールが高い統合強度を共有していれば、その結合は不均衡だ――システム全体から見て同じサービス内に存在していても。

マイクロサービスや分散システムがなくても、高距離の結合問題は起こりえる。単一のデプロイ可能ユニットを持つ一人のプロジェクトでも、モジュールレベルの抽象化においてはクロスモジュール統合が利用可能な最大距離になるため、モジュール間の密結合に苦しむことがある。

#### Lifecycle Coupling（ライフサイクル結合）: 反力

変更コストに対抗するのが **lifecycle coupling（ライフサイクル結合）** だ: コンポーネントが近いほど、それらのライフサイクルはより強く束縛される――一緒にテストおよびデプロイしなければならない。多くの「モノリスからマイクロサービスへ」のリファクタリングは主としてライフサイクル依存を減らすモチベーションから行われた。

距離は**トレードオフ**だ: 距離が低ければ共進化のコストは安いがライフサイクル結合が強まる。距離が高ければライフサイクルはより独立するが共進化のコストが高くなる。

#### 距離はソシオテクニカルである

距離は純粋に技術的ではない。**組織構造**も距離に大きく影響する:

- **同じチーム**が実装した2つのマイクロサービスは、コード構造が同一でも**異なるチーム**が実装した2つのマイクロサービスより実効距離が低い。
- 複数チームが変更を調整しなければならない場合、認知的努力とコラボレーションのオーバーヘッドが大幅に増大する。

これはコンウェイの法則を反映している: 組織構造がシステムのアーキテクチャを形成する。

#### ランタイム結合（Runtime Coupling）

コンポーネント間の**ランタイム依存関係**も距離に影響する。同期統合は非同期統合よりも強いライフサイクル結合を生む。強いランタイム依存はコンポーネントのライフサイクルを束縛する――一方の障害が他方へ連鎖することがある。

ランタイム結合を減らすこと（例: 非同期メッセージングによる）は、コンポーネント間の実効距離を増大させる。結合をバランスよく保つには、この増大した距離は統合強度を下げることで補う必要がある。

### 2.3 変動性（Volatility）

統合強度は、変更が境界を越えて伝播する**可能性**を反映する。距離は連鎖変更の**コスト**を決定する。変動性――第三の次元――は、**コンポーネントがそもそも変更を必要とする確率**を反映する。

変動性が高いほど、設計上の問題はより深刻で「痛い」ものになる。高い統合強度と高い距離を持ちながら決して変わらないコンポーネントは実際的な問題を起こさない。しかし同じコンポーネントが頻繁に変更されれば、絶え間ない苦痛の源になる。

#### DDDのサブドメインによる変動性の評価

未来を予測することは不可能だが、ドメイン駆動設計の**サブドメイン**の概念は有用なヒューリスティクスを提供する:

- **コアサブドメイン（Core subdomain）**（競争優位、「面白い問題」）→ **最高の変動性**。企業はこれを継続的に最適化・改善する。
- **サポーティングサブドメイン（Supporting subdomain）**（競争優位なし、既製品ソリューションなし、「退屈な問題」）→ **低変動性**。通常はCRUDデータ入力やETLプロセス。
- **ジェネリックサブドメイン（Generic subdomain）**（解決済みの問題、既製品ソリューション）→ **機能的変動性は低いが、実装変動性は様々**。機能自体は成熟していてめったに変わらない――解決済みの問題だ。しかし*実装*は変動的になりえる: プロバイダーの切り替え、別技術の採用、複数実装の並行稼働はすべて現実的な変更ベクターだ。例えばTTSエンジン（AWS Polly）はジェネリック機能だが、Google TTS、ElevenLabs、またはセルフホストソリューションへの切り替えは合理的な期待だ。そうした場合、プロバイダー固有の知識をカプセル化するために強力な統合コントラクトが不可欠だ。逆に、一部のジェネリックサブドメインの実装は本質的に固定される――認証・認可にAuth0を選択するということは、コントラクト設計に関わらずClerkへの切り替えが重大なイベントになることを意味する。ジェネリックサブドメインの境界を設計するときは、実装を切り替える確率や複数を同時に使用する確率を評価し、それに応じて統合コントラクトを強制する。

変動性の別の視点として、**ウォードリーマップ（Wardley Maps）**のコモディタイズ軸がある。

#### 本質的 vs. 偶発的（非）変動性

重要な区別:

- **偶発的変動性（Accidental volatility）**: 頻繁なコミットは必ずしもビジネスドメインが変動的であることを意味しない。統合強度と距離の不効果な管理という設計の貧困さが、コンポーネントの頻繁な変更を*引き起こす*ことがある。
- **偶発的非変動性（Accidental involatility）**: 一見安定したコンポーネントが本質的に安定しているとは限らない。ビジネスはそれを変更したいかもしれないが、設計の貧困なシステムの修正リスクと労力が高いためにできないでいる。

したがって、変動性はコミット履歴だけではなく、**ビジネスドメインの観点**（本質的変動性）から評価すべきだ。

## 3. フラクタル幾何学

ソフトウェア設計はフラクタルだ。オブジェクトからモジュール、サービス、システム全体の相互作用まで、異なる抽象化レベルでコンポーネントとその相互作用を設計する。Balanced Coupling モデルはこれらすべてのレベルに適用される。

ただし、統合強度レベルの意味は作業する抽象化レベルによって変わる。統合コントラクト、実装モデル、機能要件の定義は各レベルで異なる。低い抽象化レベルでの統合コントラクト（例: クラスのパブリックインターフェース）は、より高いレベルでは実装の詳細になりえる（例: サービス境界内）。同様に、同一モジュール内の2つのオブジェクトが共有するモデルは、オブジェクトレベルではモデル結合だが、システムレベルからはモジュールに完全に内部的なもの――見えず、無関係だ。

これは距離にも同様に適用される。任意の抽象化レベルにおいて、最大距離はそのレベルの境界だ。単一サービス内のモジュールを設計しているなら、クロスモジュール境界がその最大距離だ――システム観点からはそれらが同じ場所に存在していても。システム間の相互作用を設計しているなら、クロスシステム境界が最大距離だ。

したがって、モデルを適用するときは常に、どの抽象化レベルで作業しているかを確立する必要がある。これにより以下が決まる:

- **統合強度**: このレベルでコントラクト、モデル、機能依存、侵入的依存とは何か。
- **距離**: このレベルでの最低距離と最大距離は何か。

このフレーミングなしには、観察者の視点によって同じ結合関係が均衡にも不均衡にも見えてしまう。

## 4. 次元のバランスをとる

### 4.1 4つの極端な組み合わせ（強度 × 距離）

統合強度と距離の相互作用は、それらの極端な組み合わせを通じて理解できる:

| | **低距離** | **高距離** |
|---|---|---|
| **低強度** | **低凝集度（Low Cohesion）**（複雑性）— 無関係なコンポーネントが近くに配置されている。認知的負荷を増大させる。大きな泥の玉へとドリフトしがち。 | **疎結合（Loose Coupling）**（モジュラリティ）— 低強度は連鎖変更を最小化する。高距離はバランスが取れている。 |
| **高強度** | **高凝集度（High Cohesion）**（モジュラリティ）— コンポーネントは共に進化する必要があるが、低距離が連鎖変更を安価にする。 | **密結合（Tight Coupling）**（複雑性）— 頻繁な連鎖変更（高強度）が実装コストが高い（高距離）。分散モノリストへの一歩。 |

### 4.2 バランスルール

重要な洞察: **統合強度と距離が互いにバランスをとるとき（一方が高く他方が低い）、モジュラリティが生まれる。** **それらが等しいとき（両方高いまたは両方低い）、複雑性が生まれる。**

バイナリロジックで表現すると:

```
MODULARITY = STRENGTH XOR DISTANCE
COMPLEXITY = STRENGTH AND DISTANCE
```

これは「疎結合」と「高凝集度」が両方とも望ましい理由を説明している――それらは**共にバランスのとれた結合の形**であり、単に次元の値が異なるだけだ。

### 4.3 実用的なバランス（変動性の追加）

Eric Evans は「大規模システムのすべてが上手く設計されているわけではない」と観察した。**変動性**の次元が実用主義をもたらす。

結合が不均衡（高強度 + 高距離）でも、コンポーネントが**変更される見込みがない**場合、低変動性によってその影響が中和される。決して進化しない高い侵入的結合を持つレガシーシステムは許容できる――結合は技術的に「悪い」が実際には無害だ。

完全なバランス式:

```
BALANCE = (STRENGTH XOR DISTANCE) OR NOT VOLATILITY
```

これは結合が均衡なのは以下のいずれかを意味する:
- 強度と距離が互いにバランスをとっている（XOR）、**または**
- コンポーネントの変動性が低い（強度/距離の関係に関わらず）。

**注**: ブログでは説明のために簡略化されたバイナリモデルを使用している。書籍 *Balancing Coupling in Software Design* には、バイナリの極端ではなく連続値を使った、より精緻な数学モデルが含まれている。

## 5. ドメイン駆動設計との接続

Balanced Coupling モデルは DDD と深い接続を持つ。多くの DDD パターンとプラクティスが結合の次元に直接関連する:

### 戦略的設計（Strategic Design）
- **境界づけられたコンテキスト（Bounded context）** は、ビジネスドメインのモデルを共有するコンポーネントの物理的境界（中程度の距離、共有ライフサイクル）を定義する（モデル結合）。
- **境界づけられたコンテキスト統合パターン**（Shared kernel, Partnership, Conformist, Anti-corruption layer, Open-host service, Separate ways）は知識共有のレベルを決定する。アドホックなパターンはより多くの知識交換を許可し、形式的なパターンはそれを減らす。これは統合強度と距離の組織的側面の両方を反映する。
- 一部の統合パターン（Anti-corruption layer, Open-host service）は、高距離が存在するときに使用され、統合専用のコントラクトの背後に内部モデルをカプセル化する（コントラクト結合）。

### 戦術的設計（Tactical Design）
- **集約（Aggregate）** と **値オブジェクト（Value object）** のパターンは、トランザクション境界を共有する振る舞いを同じ場所に配置する（低距離）（高度な機能的結合）。これはバランスが取れている: 高強度は低距離によって相殺される。
- ビジネスロジック実装パターン（Transaction script, Active record, Domain model, Event-sourced domain model）は、ドメインの複雑性に基づいて選択される――これは変動性に直接関連する。

### サブドメインと変動性
- **コアサブドメイン** = 最高変動性 → 最も慎重な結合管理が必要
- **サポーティングサブドメイン** = 低変動性 → DDD は明示的にここでの実用的なショートカットを許可している（「大規模システムのすべてが上手く設計されているわけではない」— Eric Evans）。低変動性が不均衡な結合を中和するため
- **ジェネリックサブドメイン** = 低*機能的*変動性（問題は解決済み）だが、*実装*変動性は様々。プロバイダー切り替えや複数実装使用の確率が高い場合（例: TTSエンジン、ペイメントゲートウェイ、通知サービス）、プロバイダー固有の知識をカプセル化するために強力な統合コントラクトを強制する。切り替えコストが本質的に高くありそうにない場合（例: アイデンティティプロバイダー、コアデータベース）、実用的なショートカットは許容できる――ただし、この前提を確認し、それにコミットする前に実用的な変更ベクターをブレインストーミングすること

DDDのサブドメインは Balanced Coupling モデルにおけるコンポーネントの**変動性次元**を評価するための主要ツールだ。

## 6. まとめ: モデルの概要

```
+-----------------------------------------------------+
|            BALANCED COUPLING（均衡結合）              |
|                                                       |
|  3つの次元:                                           |
|  +---------------------+                              |
|  | 統合強度              | -> 連鎖変更の               |
|  | Integration Strength | 可能性                      |
|  | （共有される知識）     |                              |
|  |                      |                              |
|  | レベル:              |                              |
|  |  Intrusive（最高）   |                              |
|  |  Functional          |                              |
|  |  Model               |                              |
|  |  Contract（最低）    |                              |
|  +---------------------+                              |
|  +---------------------+                              |
|  | 距離                 | -> 連鎖変更の               |
|  | Distance             | コスト                      |
|  | （ソシオテクニカル）  |                              |
|  |                      |                              |
|  | 要因:                |                              |
|  |  コード構造          |                              |
|  |  組織構造            |                              |
|  |  ランタイム依存       |                              |
|  +---------------------+                              |
|  +---------------------+                              |
|  | 変動性               | -> 変更が                   |
|  | Volatility           | 発生する確率                 |
|  | （変更率）           |                              |
|  |                      |                              |
|  | 評価方法:            |                              |
|  |  DDDサブドメイン     |                              |
|  |  ウォードリーマップ  |                              |
|  |  （コミット履歴不可）|                              |
|  +---------------------+                              |
|                                                       |
|  バランスルール:                                       |
|  MODULARITY = STRENGTH XOR DISTANCE                   |
|  BALANCE = (STRENGTH XOR DISTANCE) OR NOT VOLATILITY  |
|                                                       |
|  重要な洞察:                                          |
|  疎結合と高凝集度はどちらも均衡結合の形であり、        |
|  別々の概念ではない。                                  |
+-----------------------------------------------------+
```

### コア原則

1. **結合は本質的に悪ではない。** それはシステムを機能させるものだ。目標はそれを排除することではなく、バランスをとることだ。
2. **結合は3つの次元で現れる。** 1つだけ評価すること（例: 依存関係の数）では全体像を見逃す。
3. **モジュラリティ = バランスのとれた次元。** 統合強度と距離が互いに均衡するとき、設計はモジュラだ。
4. **複雑性 = 不均衡な次元。** 強度と距離が両方高い（密結合/分散モノリスト）または両方低い（低凝集度/大きな泥の玉）とき、設計は複雑だ。
5. **変動性が実用主義をもたらす。** 低変動性は不均衡な結合を中和できる。システムのすべての部分が完璧な設計を必要とするわけではない。
6. **モデルはフラクタルだ。** メソッド、オブジェクト、パッケージ、サービス、システムのあらゆる抽象化レベルに適用される。
7. **モデルはソシオテクニカルだ。** 組織構造はコード構造と同様に距離に影響する。
8. **以前のモデルは組み込まれており、置き換えられない。** モジュール結合は統合強度レベルの基礎を提供する。コナセンスはそれらのレベル内の度合いを提供する。Balanced Coupling モデルは両方の盲点に対処する。

## ユーザーとのコミュニケーション

ユーザーがドメイン駆動設計の専門用語に精通していると仮定しないこと。変動性とサブドメイン分類を議論するときは、平易な言葉で概念を説明する:

- **コアサブドメイン** — ビジネスに競争優位をもたらすシステムの部分。組織が積極的に投資している「面白い問題」。頻繁に変化する。
- **サポーティングサブドメイン** — ビジネスが必要としているが差別化要因ではなく、既製品ソリューションも存在しないもの。通常はデータ入力、ETL、内部ツールのような単純なワークフロー。めったに変わらない。
- **ジェネリックサブドメイン** — 既製品ソリューションが存在する解決済みの問題（認証、決済処理、メール配信）。機能は成熟しているが、特定のプロバイダーや技術が変わることがある。

DDD の用語を使うが、初回使用時には常に平易な言葉での簡潔な説明を添える。「コアサブドメイン」が文脈なしで何を意味するかをユーザーが知っていると期待しないこと。
