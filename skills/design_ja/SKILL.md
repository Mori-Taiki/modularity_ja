# 高レベル設計（High-Level Design）

機能要件からモジュラな高レベルアーキテクチャを設計し、包括的な設計ドキュメントを生成する。すべてのアーキテクチャ上の決定に Balanced Coupling モデル（balanced-coupling_ja スキルからプリロード済み）を適用する。

## 入力

`$ARGUMENTS` にファイルパスが含まれている場合は、そのファイルを機能要件の入力として読む。
`$ARGUMENTS` が空またはvalid なファイルパスでない場合は、`AskUserQuestion` を使用してリクエストする。ヘッダー: 「要件」。質問: 「機能要件ファイルへのパスを入力してください。」
有効なファイルパスがあり、ファイルの読み込みに成功するまで進めない。

TaskCreate を使用してこの6ステップを追跡する: 要件の理解、モジュラアーキテクチャの設計、モジュール設計ドキュメントの作成、モジュールテスト仕様の作成、アーキテクチャドキュメントの作成、モジュラリティレビュー。

## インタラクションルール

ユーザー入力には常に `AskUserQuestion` を使用する。以下の原則に従う:

- **一度に1つの質問。** 複数の質問を1つのメッセージにまとめない。
- **選択肢方式を優先。** 説明付きの2〜4つの具体的な選択肢を提供する。オープンエンドな質問より答えやすい。
- **「その他」は自動。** ツールは常に自由記述の「その他」オプションを提供するため、追加する必要はない。
- **ヘッダーを使用。** 「承認」、「サブドメイン」、「結合」など最大12文字の短いラベル。

## プロセス

これらのステップを厳密に従う。各ステップは次に進む前に明示的なユーザー承認が必要だ。いずれかのステップで曖昧さに遭遇した場合は、`AskUserQuestion` を使ってユーザーに明確化を求めるために停止する。**決して前提を置かない。**

### ステップ1: 要件の理解

機能要件ファイルを読む。その後:

1. **機能要件を自分の言葉で再述する。** まとまりのある機能領域に整理する。
2. **結合を意識した設計のために不足している情報を発見する。** Balanced Coupling の優れた決定のために何が必要かを考える――ドメイン分類（変動性を決定）、組織構造（距離を決定）、統合パターン（強度を決定）。要件のギャップを特定する、特に:
   - コア vs サポーティング vs ジェネリックの分類が曖昧なビジネス領域――解釈を提案してユーザーに確認または修正を求める
   - モジュール境界に影響する組織的制約（チームの所有、デプロイ単位、共有インフラ）
   - 変動性が最も高くなる場所と設計努力を投資する場所に影響する戦略的方向性
   - 適切な結合強度が不明確な統合要件

   `AskUserQuestion` を使用して、各ギャップについて個別にユーザーに質問する。要件から明確なものはスキップする。分析を変えない質問はしない――すべての質問は結合決定に影響する曖昧さを解決するものであるべきだ。これらのカテゴリーに限定されない――要件がアーキテクチャ上の決定に影響する何かを曖昧にしていれば、それについて質問する。読んだ具体的な要件に質問を根拠づける。

3. **DDDサブドメインを使ってドメイン領域を分類する**（コア/サポーティング/ジェネリック）。これが変動性と設計努力を投資する場所を決定する。要件を分析し、自分で分類を提案する。テーブルとして提示する:

| サブドメイン | 分類 | 根拠 |
| ---------- | ---- | ---- |
| {領域1}   | コア | {理由} |
| {領域2}   | サポーティング | {理由} |
| {領域3}   | ジェネリック | {理由} |

次に `AskUserQuestion` を使ってユーザーに検証を求める:

| ヘッダー | 質問 | 選択肢 |
| ------- | ---- | ------ |
| サブドメイン | このサブドメイン分類は正しいですか？ | 1. **承認** — すべて正しい 2. **一部間違っている** — どれを変更するか教えます 3. **サブドメインが不足** — 記載されていない領域があります |

ユーザーが一部間違っていると言った場合は、どれを変更すべきかと正しい分類を確認する。

`AskUserQuestion` を使用して、完全な理解をユーザーに提示して検証を求める:

| ヘッダー | 質問 | 選択肢 |
| ------- | ---- | ------ |
| 承認 | この要件の理解は正しいですか？ | 1. **承認** — アーキテクチャ設計に進む 2. **修正が必要** — 何が間違っているか説明します 3. **コンテキストが不足** — 追加で伝えることがあります |

承認されるまで進めない。

### ステップ2: モジュラアーキテクチャの設計

Balanced Coupling モデルを使用して:

1. **高レベルモジュール**、それらの責務、それらの間の統合を特定する。
2. **変動性を分類する** DDDサブドメインを使用（コア = 高変動性、サポーティング/ジェネリック = 低変動性）。これが設計努力を投資する場所を決定する。
3. 各統合について**3つの次元すべて**（統合強度、距離、変動性）にわたって**結合を評価する**。
4. **バランスルールを適用する**: `BALANCE = (STRENGTH XOR DISTANCE) OR NOT VOLATILITY`
   - 高変動性コンポーネントについては、強度と距離がバランスするように統合を設計する:
     - 高い知識を共有する必要があるコンポーネント（機能/モデル結合）は同じ場所に配置する（低距離）→ 高凝集度。
     - 高い距離にあるコンポーネント（別サービス、別チーム）はコントラクトを通じて統合する（低強度）→ 疎結合。
   - 低変動性コンポーネントについては、実用的なショートカットが許容されることを認める。
5. **不均衡な結合にフラグを立てる**: 高強度 + 高距離（分散モノリストのリスク）または低強度 + 低距離（低凝集度/大きな泥の玉のリスク）。
6. 問題のある統合を再バランスする方法を具体的なアクションで**推奨する**。

結合評価テーブルをユーザーに提示する:

| 統合 | 強度 | 距離 | 変動性 | 均衡? | アクション |
| ---- | ---- | ---- | ------ | ----- | ---------- |
| A -> B | Model | 高（別サービス） | 高（コア） | No — 密結合 | 強度を下げる: APIを通じてコントラクトを導入する |

`AskUserQuestion` を使って各ステップをユーザーと進める。各ステップにはユーザーの承認が必要だ。モジュラアーキテクチャがユーザーに完全に検証されるまで設計ドキュメントの作成に進めない。

| ヘッダー | 質問 | 選択肢 |
| ------- | ---- | ------ |
| 承認 | このモジュラアーキテクチャは正しいですか？ | 1. **承認** — 設計ドキュメントに進む 2. **修正が必要** — 何を調整するか説明します 3. **再検討** — モジュール境界を再考しましょう |

### ステップ3: モジュール設計ドキュメントの作成

ステップ2の検証済みアーキテクチャを使用して、各モジュールについて `docs/design/$(date +%Y-%m-%d)/{module-name}/design.md` を作成する（以下の内容を含む）:

```markdown
# {モジュール名}

## 機能的責務（Functional Responsibilities）

このモジュールが行うこと――実装する機能とそれが提供するビジネス機能。

## カプセル化された知識（Encapsulated Knowledge）

このモジュールが知っていて、他のモジュールが知るべきでないこと――所有するドメイン概念、ビジネスルール、実装の詳細。

## サブドメイン分類（Subdomain Classification）

コア / サポーティング / ジェネリック — と分類の根拠。

## 統合コントラクト（Integration Contracts）

このモジュールが統合する各モジュールについて:

- **方向**: どのモジュールがどのモジュールに依存しているか
- **コントラクトタイプ**: 統合強度レベル（コントラクト / モデル / 機能的）
- **共有されるもの**: 交換される具体的な知識
- **コントラクト定義**: 境界を定義するインターフェース、API、イベント、またはデータ構造

## 変更ベクター（Change Vectors）

このモジュール**のみ**の変更が必要な合理的な将来の変更――このモジュールの境界が設計を支援する進化の軸。
```

個別の承認を求めずにすべてのモジュール設計ドキュメントを作成する。モジュラアーキテクチャはすでにステップ2で承認された――ドキュメントはその承認済み設計の直接的な変換だ。

すべてのモジュールドキュメントを作成した後、`AskUserQuestion` を使ってユーザーに完全なセットのレビューを求める:

| ヘッダー | 質問 | 選択肢 |
| ------- | ---- | ------ |
| モジュール | すべてのモジュール設計ドキュメントが作成されました。どうですか？ | 1. **承認** — テスト仕様に進む 2. **修正が必要** — どのモジュールに作業が必要か説明します 3. **アーキテクチャを再検討** — ドキュメントが境界の問題を明らかにした |

承認されるまで繰り返す。

### ステップ4: モジュールテスト仕様の作成

各モジュールについて、`docs/design/$(date +%Y-%m-%d)/{module-name}/tests.md` を作成する（以下の内容を含む）:

```markdown
# {モジュール名} — テスト仕様

## ユニットテスト（Unit Tests）

モジュールの内部ロジックを単独でテスト。ビジネスルール、計算、状態遷移、エッジケースをカバー。

## 統合コントラクトテスト（Integration Contract Tests）

モジュールが統合コントラクトを遵守することを検証するテスト――コントラクト定義に従った有効な入力が与えられたとき、正しい出力を生成することを確認する。

## 境界テスト（Boundary Tests）

モジュールが無効な入力を正しく拒否し、境界でのエッジケースを処理し、カプセル化を維持することを検証するテスト（何も漏洩しない）。

## 振る舞いテスト（Behavior Tests）

外部からの視点でモジュールの機能的責務を検証するテスト――ビジネスシナリオが与えられたとき、モジュールが期待通りに動作することを確認する。
```

各テストセクションには、以下の具体的な名前付きテストケースを含める:

- **テスト名**: 説明的な名前
- **シナリオ**: 何をテストしているか
- **期待される振る舞い**: 正しい結果は何か

承認を求めずにすべてのテスト仕様を作成する。モジュール設計はすでに承認された――テスト仕様はそれらから直接導出される。

### ステップ5: アーキテクチャドキュメントの作成

`docs/design/$(date +%Y-%m-%d)/architecture.md` を作成する（以下の内容を含む）:

```markdown
# アーキテクチャ概要（Architecture Overview）

## 機能要件サマリー（Functional Requirements Summary）

このアーキテクチャが対処する要件の簡潔なまとめ。

## モジュールマップ（Module Map）

すべてのモジュールのリストと1行の説明。

## モジュールの連携方法（How the Modules Work Together）

各主要な機能フロー / ユースケースについて:

- どのモジュールが参加するか
- データ/制御がそれらの間をどのように流れるか
- 相互作用を管理するコントラクトは何か

## 結合評価（Coupling Assessment）

Balanced Coupling モデルに根拠づけられた主要な設計決定とその根拠についての解説を伴う、モジュラアーキテクチャ分析からの結合評価テーブル。

## 設計決定とトレードオフ（Design Decisions and Trade-offs）

主要なアーキテクチャ上の決定、考慮されたこと、選ばれたこと、その理由――結合の次元とバランスルールに根拠づけて。

## 未解決のリスク（Unresolved Risks）

設計が意図的に開いたままにしていること、および再検討すべき条件。
```

承認を求めずにアーキテクチャドキュメントを作成する。すでに承認されたモジュール設計を統合する。

### ステップ6: モジュラリティレビュー

すべてのドキュメントが作成された後、モジュラリティ不均衡について自身の設計をレビューする。モジュール間の各統合について:

1. **統合をマッピングする**: 何の知識が共有されているか？暗黙的か明示的か？統合強度レベルは何か？
2. **3つの次元すべてを評価する**: 強度、距離、変動性。
3. **バランスルールを適用する**: `BALANCE = (STRENGTH XOR DISTANCE) OR NOT VOLATILITY`
4. **不均衡にフラグを立てる**: 不均衡かつ変動性が高い統合に焦点を当てる。

見つかった各問題について、重大度を分類する:

- **Critical（重大）**: 高強度 + 高距離 + 高変動性。
- **Significant（重要）**: 中程度の変動性領域での不均衡な結合、または統合ポイントを隠す暗黙的な結合。
- **Minor（軽微）**: 低変動性領域での不均衡な結合、または認知的負荷を増やすが連鎖変更を引き起こさない低凝集度。

Critical または Significant の問題がある場合:

1. ステップ2の結合評価テーブルと同じ構造に加え、各問題の知識の漏洩の説明と推奨改善点を含む形で、ユーザーに問題を提示する。
2. 結合を再バランスするための設計への具体的な変更を提案する。
3. ユーザーが変更を承認したら、影響を受けた設計ドキュメント（モジュール設計、テスト仕様、アーキテクチャドキュメント）を更新する。
4. **このステップを繰り返す** — 更新された設計を再度レビューする。Critical または Significant の問題がなくなるまで続ける。

Minor の問題はアーキテクチャドキュメントの「未解決のリスク」セクションに記載するが、完了をブロックしない。

## 重要な制約

- **決して前提を置かない。** 何か不明な場合は、進める前にユーザーに質問する。
- **実行可能なドキュメントを作成する。** すべての設計ドキュメントは開発者が実装を始めるのに十分なコンテキストを提供すること。すべてのテスト仕様は開発者がテストを書き始めるのに十分なコンテキストを提供すること。
- **「すべてを分離すること」を推奨しない。** それは強度と距離のバランスを無視する。
- **1つの次元だけで結合を評価しない。** 常に3つすべてを考慮する: 強度、距離、変動性。
- **本質的変動性と偶発的変動性を区別すること。** 高いコミット頻度は設計の貧困さ（偶発的変動性）を示す場合があり、変動的なドメインを示すわけではない。
- **距離の組織的次元を考慮すること。** 同じコード構造 + 別のチーム = より高い実効距離。
- **すべての推奨事項をモデルに根拠づけること。** 各設計決定を正当化する具体的な次元と原則を参照する。
