# モジュラリティレビュー

Vlad Khononov による Balanced Coupling モデル（balanced-coupling_ja スキルからプリロード済み）を使用してコードベースのモジュラリティ不均衡を分析する。知識のカプセル化、複雑性、連鎖変更の観点から具体的な設計上の問題を特定し、設計を改善する方法を説明するレビューを生成する。

TaskCreate を使用してこの4ステップを追跡する: 問題ドメインの理解、統合のマッピング、バランスルールの適用、レビューの作成。

## インタラクションルール

ユーザー入力には常に `AskUserQuestion` を使用する。以下の原則に従う:

- **一度に1つの質問。** 複数の質問を1つのメッセージにまとめない。
- **選択肢方式を優先。** 2〜4つの具体的な選択肢を提供する。オープンエンドな質問より答えやすい。
- **「その他」は自動。** ツールは常に自由記述の「その他」オプションを提供する――手動で追加しない。
- **ヘッダーを使用。** 「スコープ」、「ドメイン」、「チーム」、「問題点」など最大12文字の短いラベル。

## プロセス

### ステップ1: 問題ドメインの理解

1. `AskUserQuestion` を使用して、分析するコードベースの部分を確認する。ヘッダー: 「スコープ」。選択肢: 「コードベース全体 — すべてのコンポーネントを分析」、「特定のディレクトリ — パスを指定します」、「特定のコンポーネント — 名前を指定します」。特定のスコープを選択した場合は、詳細を収集するためにフォローアップする。

2. **読んでから質問する。** `docs/` フォルダ内のすべての機能要件ドキュメントを読み、次にコード自体を読む。コンポーネント、それらの責務、統合方法を理解する。ナビゲートには LSP（findReferences, goToDefinition）、Grep、Glob を使用する――推測しない。

3. **理解を提示する。** ドメインに関する質問をする前に、コードと要件から学んだことの簡潔な要約を提示する:
   - 発見したコンポーネントとそれらの責務
   - 観察した統合パターン（共有型、API呼び出し、データベースアクセス、イベントフロー）
   - ドメイン分類（コア/サポーティング/ジェネリック）の最善の推測と根拠、確信度レベル――確信度が低い領域がフォローアップ質問の最有力候補
   - チーム構造、デプロイトポロジー、設計意図についての前提

   `AskUserQuestion` を使用して検証する。ヘッダー: 「要約」。選択肢: 「概ね正しい」、「一部間違っている点がある」、「重要なコンテキストが抜けている」。ユーザーが修正や追加情報を提供した場合は、次に進む前にそれを取り込む。

4. **まだ必要な情報を見つける。** Balanced Coupling モデルを知っている。変動性（ドメイン分類から）、距離（組織構造から）、強度（コードから）が必要なことを知っている。もしわかっていたら結合評価を変えるものを考え、そのギャップについて質問する。`AskUserQuestion` を使って一度に1つずつ。分析を変えない質問はしない――すべての質問は評価に重要なギャップを埋めるものであるべきだ。

   考慮すべき一般的な情報ギャップ（コード、要件、またはユーザーの修正からすでに答えられるものはスキップ）:
   - **ドメイン分類のギャップ** — あるものがコア（競争優位、高変動性）かサポーティングかジェネリックかが判断できない領域。最善の推測を提案し、ユーザーに確認または修正を求める。
   - **組織的コンテキスト** — チームの所有境界、デプロイトポロジー、共有インフラ。コード構造が示す以上に実効距離に影響する。
   - **既知の問題点** — 変更が予想外にコストがかかる領域、デプロイが問題を起こす場所、設計に違和感を感じる領域。これらは最も重要な場所への分析を焦点化する。
   - **戦略的方向性** — 変動性が高い領域に影響する今後のマイグレーション、ビジネス転換、または計画された変更。
   - **驚くべきパターン** — コードで見つけたもので、意図的な設計選択か偶発的な複雑性か判断できないもの。前提を置く前に質問する。

   これらのカテゴリーに限定されない。コードで見つけたもので、適切な結合評価のために明確化が必要なものがあれば、質問する。具体的なコードの観察に基づいて質問を根拠づける――実際に見つけたコンポーネント、パターン、統合を参照する。

### ステップ2: 統合のマッピング

相互作用するコンポーネントの各ペアについて、以下を特定する:

- **共有される知識** — 実装の詳細、ビジネスルール、ドメインモデル、または統合コントラクト？
- **統合強度レベル** — Intrusive（侵入的）、Functional（機能的）、Model（モデル）、Contract（コントラクト）結合？
- **共有される知識は暗黙的か明示的か？** 暗黙的な結合（重複したビジネスルール、直接データベースアクセス、内部動作への前提）は特に危険だ。
- **距離** — 同じモジュール、同じサービス、別サービス、別システム？同じチームか別のチームか？同期か非同期か？
- **変動性** — ビジネスドメインの観点から、この領域がどれくらい変わりそうか？ジェネリックサブドメインについては、機能的変動性（問題の定義）と実装変動性（特定のプロバイダー/技術）を区別する。

### ステップ3: バランスルールの適用

各統合について適用する: `BALANCE = (STRENGTH XOR DISTANCE) OR NOT VOLATILITY`

結合が**不均衡かつ変動性が高い**すべての統合にフラグを立てる:

- **高強度 + 高距離 + 高変動性** — 変動性の高い領域の密結合。緊急の問題。変更は頻繁で、コストが高く、予測不可能になる。
- **低強度 + 低距離** — 潜在的な低凝集度。無関係なコンポーネントが同じ場所に配置され、認知的負荷が増大し、大きな泥の玉へとドリフトする。
- **高強度 + 高距離 + 低変動性** — 技術的負債だが許容できる。注記するが優先しない。

### ステップ4: レビューの作成

document_ja スキル（プリロード済み）を使用して、Markdown と HTML の両形式でモジュラリティレビューを生成する。document_ja スキルが構造と出力形式を定義する。

## 重要な制約

- **コードを読むこと。** 構造だけで問題を特定しない。実際の統合ポイントを読む――関数呼び出し、インポート、共有データ構造、データベースアクセスパターン、API呼び出し――実際に何の知識が共有されているかを判断するために。
- **1つの次元だけで結合を評価しない。** 常に3つすべてを考慮する: 強度、距離、変動性。
- **本質的変動性と偶発的変動性を区別すること。** 高いコミット頻度は設計の貧困さ（偶発的変動性）を示す場合があり、変動的なドメインを示すわけではない。変動性はビジネスドメインの観点から評価する。
- **すべてにフラグを立てない。** 不均衡かつ変動性が高い統合に焦点を当てる。30の軽微な問題にフラグを立てるレビューは、明確な説明を伴う5つの重大な問題を特定するレビューより有用でない。
- **すべての問題をモデルに根拠づけること。** 統合を問題にする特定の結合次元、強度レベル、またはバランスルールの原則を参照する。
- **「すべてを分離すること」を推奨しない。** 分解は距離を増大させる。強度がすでに増大した距離を支えるほど低いとき、またはライフサイクル結合が主要なボトルネックであるときにのみ推奨する。
- **組織的次元を考慮すること。** 同じコード構造 + 別のチーム = より高い実効距離。それが分析に影響する場合は、チームの所有について確認する。
